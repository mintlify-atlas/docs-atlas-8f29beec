## Endpoint

```
PATCH /tickets/:id
```

Updates an existing ticket. Use this endpoint to change the ticket status, assign technicians, update priority, or modify other ticket properties.

## Authentication

<Note>
This endpoint requires JWT Bearer authentication.
</Note>

## Path Parameters

<ParamField path="id" type="string" required>
  Unique identifier of the ticket to update
  
  **Format:** UUID v4
  
  **Example:** `123e4567-e89b-12d3-a456-426614174000`
</ParamField>

## Request Body

<Note>
All fields are optional. Only include the fields you want to update.
</Note>

<ParamField body="status" type="enum">
  New status for the ticket
  
  **Possible values:**
  - `open` - Newly created or reopened
  - `in_progress` - Being actively worked on
  - `resolved` - Issue has been resolved
  - `closed` - Ticket is closed and archived
  
  **Example:** `"in_progress"`
</ParamField>

<ParamField body="priority" type="enum">
  Updated priority level
  
  **Possible values:**
  - `low` - Low priority
  - `medium` - Medium priority
  - `high` - High priority
  
  **Example:** `"high"`
</ParamField>

<ParamField body="technicianId" type="string">
  ID of the technician to assign to the ticket
  
  **Format:** UUID v4
  
  **Example:** `"789e1234-e89b-12d3-a456-426614174002"`
</ParamField>

<ParamField body="comment" type="string">
  Update comment explaining the changes
  
  **Constraints:**
  - Maximum length: 500 characters
  
  **Example:** `"Se asigna al técnico para revisión"`
</ParamField>

<ParamField body="title" type="string">
  Updated title for the ticket
  
  **Constraints:**
  - Minimum length: 5 characters
  - Maximum length: 100 characters
</ParamField>

<ParamField body="description" type="string">
  Updated description for the ticket
  
  **Constraints:**
  - Minimum length: 10 characters
</ParamField>

<ParamField body="categoryId" type="string">
  Updated category ID
  
  **Format:** UUID v4
</ParamField>

## Status Transitions

<Warning>
Not all status transitions are valid. Attempting invalid transitions will result in a 400 Bad Request error.
</Warning>

### Valid Transitions

| From | To | Description |
|------|-----|-------------|
| `open` | `in_progress` | Technician starts working on ticket |
| `open` | `resolved` | Quick resolution without in-progress state |
| `in_progress` | `resolved` | Ticket is resolved |
| `in_progress` | `open` | Ticket needs reassignment |
| `resolved` | `closed` | Client confirms resolution |
| `resolved` | `open` | Issue persists, ticket reopened |
| `closed` | `open` | Reopen closed ticket |

### Invalid Transitions

<Warning>
- **open → closed**: Cannot close without resolution
- **in_progress → closed**: Must resolve before closing
</Warning>

## Validation Rules

- **status**: Must be one of the valid enum values
- **priority**: Must be one of the valid enum values
- **technicianId**: Must be a valid UUID v4 format
- **comment**: Maximum 500 characters
- **title**: 5-100 characters if provided
- **description**: Minimum 10 characters if provided
- **categoryId**: Must be a valid UUID v4 format if provided
