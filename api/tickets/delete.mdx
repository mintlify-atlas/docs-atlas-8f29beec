## Endpoint

```
DELETE /tickets/:id
```

Permanently removes a ticket from the system. This action is irreversible and should be used with caution.

<Warning>
This endpoint is restricted to administrators only. Regular users and technicians cannot delete tickets.
</Warning>

## Authentication

<Note>
This endpoint requires JWT Bearer authentication with administrator privileges.
</Note>

## Path Parameters

<ParamField path="id" type="string" required>
  Unique identifier of the ticket to delete
  
  **Format:** UUID v4
  
  **Example:** `123e4567-e89b-12d3-a456-426614174000`
</ParamField>

## Response

Returns a success message upon successful deletion.

<ResponseField name="message" type="string">
  Confirmation message
  
  **Example:** `"Ticket eliminado exitosamente"`
</ResponseField>

## Example Request

<CodeGroup>

```bash cURL
curl -X DELETE https://api.example.com/tickets/123e4567-e89b-12d3-a456-426614174000 \
  -H "Authorization: Bearer YOUR_ADMIN_JWT_TOKEN"
```

```javascript JavaScript
const response = await fetch(
  'https://api.example.com/tickets/123e4567-e89b-12d3-a456-426614174000',
  {
    method: 'DELETE',
    headers: {
      'Authorization': 'Bearer YOUR_ADMIN_JWT_TOKEN'
    }
  }
);

const result = await response.json();
console.log(result.message);
```

```python Python
import requests

ticket_id = '123e4567-e89b-12d3-a456-426614174000'
url = f'https://api.example.com/tickets/{ticket_id}'
headers = {'Authorization': 'Bearer YOUR_ADMIN_JWT_TOKEN'}

response = requests.delete(url, headers=headers)
result = response.json()
print(result['message'])
```

</CodeGroup>

## Authorization Requirements

<Warning>
Only users with administrator role can delete tickets. Attempting to delete a ticket without proper permissions will result in a 403 Forbidden error.
</Warning>

### Permission Hierarchy

| Role | Can Delete Tickets |
|------|-------------------|
| Admin | Yes |
| Technician | No |
| Client | No |

## When to Delete Tickets

<Note>
Deletion should be used sparingly. Consider these alternatives:
</Note>

### Use Deletion For:
- Test or duplicate tickets
- Spam or malicious tickets
- Data cleanup during maintenance
- Compliance with data deletion requests

### Consider Alternatives:
- **Closing tickets**: Use status `closed` instead of deletion for resolved issues
- **Archiving**: Implement soft deletion or archiving instead of permanent deletion
- **Status changes**: Update status to `closed` to maintain audit trail

## Best Practices

1. **Audit Trail**: Log all ticket deletions for compliance and auditing purposes
2. **Confirmation**: Always confirm with the user before deleting a ticket
3. **Backup**: Ensure ticket data is backed up before deletion
4. **Alternative Actions**: Prefer closing tickets over deletion when possible
5. **Permissions**: Restrict deletion to administrators only

## Cascading Effects

<Warning>
Deleting a ticket may affect related data:
</Warning>

- Associated comments or updates may be deleted
- History and audit logs may be removed
- Related notifications may be affected
- Analytics and reporting data may be impacted

## Implementation Notes

- Deletion is **permanent** and **irreversible**
- The endpoint validates that the user has administrator privileges
- UUID format is validated before attempting deletion
- Returns 404 if the ticket doesn't exist
- Returns 403 if the user lacks deletion permissions

## Alternative: Soft Delete

<Note>
Consider implementing soft deletion in your application:
</Note>

```json
// Instead of DELETE, use PATCH with a deleted flag
PATCH /tickets/:id
{
  "deleted": true,
  "deletedAt": "2024-03-15T12:00:00.000Z",
  "deletedBy": "admin-user-id"
}
```

Soft deletion allows:
- Recovery of accidentally deleted tickets
- Maintaining audit trails
- Compliance with data retention policies
- Better analytics and reporting
